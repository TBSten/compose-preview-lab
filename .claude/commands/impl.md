# Implementation Task Command

This command helps you implement a feature or fix from start to finish.

## フロー

### 1. 実装内容を把握する

実装内容は以下の通り。これを把握して、タスク名を英語で考える。

```````md
$1
```````

### 2. 期待されるアウトプット・検証方法を考えてまとめる

期待されるアウトプットと検証方法を以下のフォーマットで `.local/task-{タスク名}/01_期待されるアウトプットと検証方法.md`
ファイルを作成する。

```md

# 期待されるアウトプットと検証方法 {タスク名}

## 期待されるアウトプット

{期待されるアウトプット}

## 検証方法

{検証方法}

```

- `{期待されるアウトプット}` にはタスク実装後できるようになる機能の詳細を簡潔に記載する。
- `{検証方法}` にはタスク実装後、どのようにテストするかを具体的に記載する。
    - integrationTest や dev モジュールを積極的に使って検証するように努める。
    - 明らかに不必要と考えらる場合以外は、必ず 「`.github/workflows/pull-request.yml` を参考に、CI をローカルでエミュレートする。」を追加する。
    - その他タスクの内容に応じてファイルを確認したり・コマンドを実行する・テストコードを作成して実行するなどの検証方法を考えて記載する。
    - 例:
        - `integrationTest/app/src/commonTest/` にテストコードを用意し、`./gradlew jvmTest` を実行してテストが成功することを確認する。
        - `integrationTest/app/build/generated/ksp/jvm/jvmMain/kotlin` に出力された自動生成ソースのファイル内容を確認する。
        - `.github/workflows/pull-request.yml` を参考に、CI をローカルでエミュレートする。

### 3. 実装する

### 4. 実装された内容のサマリーを `.local/task-{タスク名}/02_実装内容サマリー.md` に出力する

```md
# 実装内容 {タスク名}

$1

## 実装内容サマリー

{実装内容サマリー}

## 実装していないこと

{実装していないこと}
```

- `{実装内容サマリー}` に簡単に説明と重要なファイル・差分内容を記載。
- `{実装していないこと}` に今回は実装しなかったが、考えられる追加機能などを記載する。そのままコピペして AI Agent
  に共有できるように簡潔に多くても200文字程度で、いくつかリストアップする。
